Purplle Retail Intelligence Documentation Center

Introduction

Purplle Retail Intelligence is an end-to-end computer vision and retail analytics application. It converts CCTV or MP4 store footage into structured visitor events, saves those events through a FastAPI backend, and presents store KPIs in a browser dashboard.

The system helps physical retail teams answer questions that are easy online but hard in stores: How many real customers entered? Which zones attracted attention? Where did shoppers drop out? Is the billing queue becoming a problem? Are staff members polluting customer counts?

Core Features

  • Person detection and tracking using YOLO11n and tracker IDs.
  • Store layout mapping with zone polygons, camera roles, and optional homography calibration.
  • Idempotent event ingestion through POST /events/ingest.
  • Dashboard KPIs: unique visitors, conversion rate, average dwell, queue depth, and abandonment rate.
  • Funnel, heatmap, recent event feed, per-camera stats, system telemetry, and anomaly cards.
  • POS CSV loading and conversion matching within the analytics layer.
  • Raw and annotated video streaming with HTTP range support.
  • Docker-first runtime with health, readiness, and liveness probes.

Key Differentiators

Business readable, engineering inspectable. The same application serves store leaders who need decisions and engineers who need raw event detail, traceability, and testable API contracts.

First Workflow

  1. Open http://localhost:8000/dashboard.
  2. Choose a store and camera from the dashboard controls.
  3. Use the video simulation panel to process an available clip.
  4. Watch the simulation progress until it finishes or reports cache reuse.
  5. Review KPI cards, funnel, heatmap, anomalies, and recent events.
  6. Open http://localhost:8000/guide for this documentation and http://localhost:8000/docs for Swagger.

Example Use Case

A manager wants to know if a billing queue is hurting conversion. They open the dashboard, filter to the billing camera, run or inspect the billing clip, check Current Queue Depth, inspect Abandonment Rate, and review anomaly cards. If a queue spike appears, the suggested action is to open another billing counter or assign staff to checkout.

Feature Documentation

Event Ingestion

Overview: The API accepts batches of visitor events, validates them with Pydantic, normalizes event type aliases, and writes idempotently by event_id.

Workflow: Pipeline detects movement, emits JSON, posts to /events/ingest, database stores valid rows, failed rows are reported in the same response.

Inputs: Array of event objects matching EventSchema.

Outputs: {"ingested": n, "failed": n, "errors": [...]}.

Configuration: MAX_INGEST_BATCH, RATE_LIMIT_PER_MINUTE, INGEST_URL.

FAQ: Duplicate event IDs count as successful ingestion because the endpoint is idempotent.

Metrics

Overview: Computes unique visitors, conversion rate, average dwell, dwell per zone, current queue depth, and abandonment rate.

Workflow: Filters out staff, groups by visitor ID, detects billing events, matches POS timestamps within a five-minute window, and calculates rates.

Inputs: Store ID and optional camera ID.

Outputs: JSON KPI object for dashboard cards.

Limitations: Conversion depends on POS data being loaded and timestamp alignment being reasonable.

Funnel

Overview: Tracks progression from entry to retail zone visit to billing queue to purchase.

Workflow: Entry camera visitors form the base. Zone visits and billing events advance users through stages. POS matching indicates purchase.

Outputs: Stage counts and dropoff percentages.

Heatmap

Overview: Summarizes zone engagement by visits, average dwell seconds, and relative intensity.

Workflow: Reads non-staff events with zone IDs, excludes entry and exit zones, aggregates by zone, and computes intensity against the busiest zone.

Limitations: data_confidence becomes true only after at least 20 unique sessions.

Anomaly Detection

Overview: Produces operational warnings for queue spikes, conversion drops, and dead zones.

AnomalyTriggerAction
STATISTICAL_QUEUE_SPIKECurrent queue depth exceeds a seven-day rolling threshold when enough queue history exists.Open additional billing counter.
BILLING_QUEUE_SPIKEFallback threshold: queue depth at least 5 warns, at least 8 is critical.Monitor or immediately add checkout capacity.
CONVERSION_DROPCurrent conversion is more than 30 percent below seven-day baseline, or low with enough traffic when no baseline exists.Check checkout bottlenecks and staff availability.
DEAD_ZONERetail zone receives zero visits in the recent 30-minute window while the store has traffic.Inspect visibility, display, stock, or signage.

POS Loading

Overview: Loads CSV transaction data from project-root files matching POS*.csv, pos*.csv, or *transactions*.csv.

Workflow: Call POST /api/load-pos. The loader parses actual Purplle-style columns and filters zero-amount loyalty card scans.

Outputs: Per-file processing result.

Video Streaming and Simulation

Overview: The dashboard lists available MP4 clips, streams them with range requests, checks for annotated videos, and can start background detection.

Limitations: Safe path resolution strips directory components for stream endpoints. Duplicate filenames across directories can be ambiguous for direct stream calls.

Event Ingest

URL: POST /events/ingest

Purpose: Store structured visitor events. The endpoint returns HTTP 207 Multi-Status because individual records can succeed or fail within one batch.

Request example:

curl -X POST http://localhost:8000/events/ingest \
  -H "Content-Type: application/json" \
  -d '[
    {
      "event_id": "4c2bde79-ff5f-4e8a-9c42-bfd4f0bfe5a8",
      "store_id": "ST1008",
      "camera_id": "cam1",
      "visitor_id": "VIS_001",
      "event_type": "ENTRY",
      "timestamp": "2026-06-09T10:00:00Z",
      "zone_id": "ENTRY",
      "dwell_ms": 0,
      "is_staff": false,
      "confidence": 0.93,
      "metadata": {
        "queue_depth": null,
        "sku_zone": null,
        "session_seq": 1
      }
    }
  ]'

Response example:

{
  "ingested": 1,
  "failed": 0,
  "errors": []
}

Error responses:

StatusWhenExample Detail
413Batch exceeds hard or configured maximum.Batch too large. Max supported events per request is 500.
207Some records fail schema validation or monotonicity.Errors array contains per-event failures.
429Rate limiter enabled and caller exceeds limit.Rate limit exceeded plus Retry-After.

Infrastructure

The default infrastructure is a single Dockerized FastAPI service exposing HTTP. It mounts the repository into /workspace, uses SQLite by default, and can process local video files. For multi-store production, migrate persistence to a managed database, place the service behind HTTPS and auth, and move CV processing to controlled edge workers or job runners.

Authentication and Authorization

The app now supports lightweight API-key protection for write/admin endpoints. It is intentionally optional so demos and tests remain simple. For public or enterprise deployments, use API_KEY at minimum and add one of the following before exposing the app broadly:

  • Private network only, with VPN or zero-trust access.
  • Reverse proxy authentication in front of FastAPI.
  • JWT/OAuth/OIDC middleware with role checks for write endpoints.
  • IP allowlists for CV pipeline producers.

Data Protection

Events are PII-adjacent because they describe behavior, movement, and possible demographic estimates. Use least-privilege database credentials, encrypted disks, short retention windows, anonymized visitor IDs, and clear policies for footage storage.

Session Management

The current app does not manage user login sessions. Visitor sessions are analytical sessions represented by visitor_id and metadata.session_seq.

Cloud Deployment

The Dockerfile listens on ${PORT} and defaults to 8080, which fits platforms that inject PORT. For a cloud service, set environment variables explicitly, attach durable storage or a managed database, and put the service behind HTTPS.

For Hugging Face Spaces, configure APP_PUBLIC_BASE_URL to the public Space URL, keep secrets in Space secrets, and set API_KEY before exposing write endpoints publicly. If using the previous deployment URL, the canonical URL should point to the active production Space or custom domain.

Rollback Steps

  1. Stop new pipeline jobs or point them to the previous API URL.
  2. Redeploy the last known good container image.
  3. Restore the previous database snapshot if schema or data changes caused corruption.
  4. Validate /health, /ready, dashboard load, and one metrics endpoint.
  5. Resume pipeline jobs after confirming event ingest works.

Dashboards and Alerts

No vendor-specific dashboard is bundled. Recommended production setup: collect stdout logs, export FastAPI metrics, add uptime checks for /live and /ready, and route critical anomalies to store support channels.

FAQ

1. What does the app do?

It converts CCTV or MP4 footage into retail events and dashboard KPIs.

2. Where do I start?

Run Docker Compose, open /dashboard, and process a sample video.

3. Is there a login?

There is no user login yet. Production write endpoints can be protected with API_KEY; use trusted networks or an auth proxy for full user authentication.

4. Where is Swagger?

Open /docs.

5. Where is this guide served?

Open /guide.

6. What stores are configured?

ST1008 Brigade Road Bangalore and ST1076 Purplle MUM 1076.

7. How are staff excluded?

Metrics filter events where is_staff is false.

8. Why is conversion zero?

POS data or billing events may be missing, or timestamps may not match within the five-minute window.

9. Can I ingest duplicate events?

Yes. Duplicate event_id values are treated as successful idempotent inserts.

10. What timestamp format is required?

Use strict UTC ISO format such as 2026-06-09T10:00:00Z.

11. Can I run without Docker?

Yes, install Python dependencies and run Uvicorn locally.

12. Why does data reset?

RESET_DB_ON_STARTUP defaults to 1 for local stateless runs.

13. How do I preserve data?

Set RESET_DB_ON_STARTUP=0 and back up the SQLite file.

14. What is the maximum ingest batch?

Default and hard limit are 500 events unless configured lower.

15. What is rate limiting?

An optional in-memory per-IP limiter controlled by RATE_LIMIT_PER_MINUTE.

16. Does the dashboard use WebSockets?

No. It polls REST endpoints.

17. How are videos streamed?

Through FastAPI file responses with HTTP range support.

18. How do annotated videos appear?

The app checks for files prefixed with annotated_.

19. What is a dead zone?

A retail zone with no recent customer visits while the store has traffic.

20. What is queue spike severity?

It depends on rolling queue history or fallback thresholds.

21. Can I add a new store?

Yes. Add a store entry, cameras, roles, folders, floor plan, and zones to store_layout.json.

22. Can I add a new event type?

Yes, but update app/models.py, analytics logic, tests, and docs.

23. What database is used?

SQLite by default through SQLAlchemy.

24. Is PostgreSQL supported?

SQLAlchemy can point to another database URL, but the project is currently documented and tested around SQLite.

25. How do I run tests?

Use python3 -m pytest tests/ -v or ./run-tests.sh.

26. What quality tools are used?

Ruff, mypy, pytest, pytest-cov, and pre-commit are configured.

27. Where are logs?

Structured JSON logs are written to stdout.

28. How do I report security issues?

Use the process in SECURITY.md; do not open public issues for vulnerabilities.

29. How often is support reviewed?

SUPPORT.md states issues are reviewed weekly on a best-effort basis.

30. What is planned next?

Roadmap includes PostgreSQL multi-tenancy, Redis pub/sub, improved calibration, and alerting.

31. How is SEO handled?

The app serves /, /robots.txt, /sitemap.xml, /site.webmanifest, and author metadata for Keshab Kumar @keshabkjha.

32. Where are official creator links exposed?

They are visible on the landing page and guide, and machine-readable from /api/project-profile.
